3. Backend Unit Tests (122 tests – engine + auth)

3.1 Simulation Engine – engine.test.ts (107 tests)

This is the most critical test file. The simulation engine contains all game logic as pure functions with no side effects, making it highly testable. Every function exported from sim/engine.ts is covered.

Utility Functions

  • clampMetric (4 tests) – Boundary clamping at 0 and 100, rounding, pass-through for valid values
  • buildInitialState (4 tests) – 8×8 grid creation, 5 fixed rock tiles at expected coordinates, plain terrain defaults, all infrastructure counters at zero
  • drawProjects (4 tests) – Always returns 2 unique projects, seed variation produces different combinations, all from valid catalogue
  • drawEvent (5 tests) – Difficulty-aware turn gating (normal: turn 3, hard: turn 2), seed-based roll vs. event chance, event structure validation, eligibility filtering
  • calculateScore (4 tests) – Weighted formula verification (happiness×400, envHealth×400, economy×200, -carbon×0.05, +budget×0.001, +turns×500), non-negative guarantee, equal weight for happiness/envHealth, 500 points per turn
  • scoreTier (6 tests) – All 6 tiers from S (≥95k) to F (<25k) with boundary values
  • achievementsForScore (9 tests) – first_win always awarded, green_city, budget_master, happy_citizens, urban_planner, hard_mode_win, carbon_neutral, survivor, difficulty-specific exclusions
  • summarizeMetrics (1 test) – Extracts all 6 metrics from session record
  • listCatalog (4 tests) – Returns all zones, infrastructure, and projects with correct structure
  • Game constants (5 tests) – MAX_TURNS = 15, MAX_ACTIONS_PER_TURN = 3, positive costs, wind turbine cap = 3

resolveTurn – Full Turn Resolution Pipeline

  • Session validation (5 tests) – Rejects non-active/lost sessions, rejects >3 actions, accepts exactly 3, allows 0 actions (skip turn)
  • Passive turn effects (7 tests) – Base budget grant + tax, population growth (happiness ≥ 60), population decline (happiness < 30), no change in between, carbon > 5000 envHealth penalty, envHealth ≥ 80 happiness bonus, passive record always first
  • Zone placement (11 tests) – Residential on empty tile, cost deduction, metric deltas (green_space with road + bike lane prereqs), rock tile rejection, occupied tile rejection, commercial requires 2 roads, industrial requires roads + waste, allows commercial with roads, unknown code rejection, insufficient budget, invalid tile index
  • Infrastructure placement (10 tests) – Road network placement + counter increment, transit counter (needs 2 roads), bike lane requires roads, water treatment requires population > 25k, wind turbine cap (3), allows wind turbine under cap, occupied tile rejection, bike lane +2 happiness bonus when envHealth ≥ 60, waste management +2 envHealth bonus, unknown code rejection
  • Project decisions (5 tests) – Approve deltas + cost deduction, reject penalty + zero cost, insufficient budget rejection, requires both decisions, skips when no pending projects
  • Event responses (5 tests) – Option deltas applied, budgetDelta applied, missing response rejection, out-of-bounds index rejection, skips when no pending event
  • Loss conditions (5 tests) – city_revolt (happiness ≤ 0), ecological_collapse (envHealth ≤ 0), bankruptcy (economy ≤ 0), no final score on loss, no turn advance on loss
  • Game completion (6 tests) – Completes on turn 15, calculates final score, assigns tier, awards achievements, no turn advance, no new projects/events drawn
  • Turn progression (3 tests) – Advances turn by 1, draws new projects, records all actions
  • Difficulty settings (4 tests) – Easy has higher starting budget, hard has lower budget, hard has earlier event start, hard has higher event chance
  • Game constants (5 tests) – MAX_TURNS = 15, MAX_ACTIONS_PER_TURN = 3, positive zone/infrastructure costs, wind turbine cap = 3

3.2 Auth Module – auth.test.ts (15 tests)

  • hashPassword (2 tests) – Returns bcrypt hash string (starts with $2), different hashes for same input (salt)

6.5 Why Cypress Tests Target localhost, Not the Live Vercel URL

Cypress E2E tests are intentionally configured to run against http://localhost:3000 (local dev stack), not the deployed Vercel URL (https://eco-city-builder-org.vercel.app).

There are several important reasons for this architectural decision:

ReasonExplanation
Database isolation E2E tests create test users, game sessions, and submit turns that write to the database. Running against the live Vercel deployment would pollute the production database with test data (e.g. hundreds of testuser_1710XXXXXX accounts), which is unacceptable.
Deterministic state Tests expect a clean, predictable starting state. A shared production database could have other users’ sessions, conflicting data, or resource limits that cause non-deterministic test failures (flaky tests).
CORS restrictions The deployed backend on Render is configured with CORS_ORIGIN set to the Vercel frontend domain. Cypress runs from localhost, so API requests from Cypress to the live Render backend would be blocked by CORS policy.
API rate limits & costs Running 45+ E2E tests repeatedly against a hosted backend (Render) consumes free-tier quotas and could trigger rate limiting, making tests unreliable.
Speed Localhost communication has near-zero latency. Running against a remote server adds network round-trip time to every Cypress command, significantly slowing the test suite.
Industry standard practice E2E tests are designed to validate application behaviour, not infrastructure. The standard approach is to run them against a local or staging environment, never directly against production.

If needed in the future, a dedicated staging environment (separate Vercel preview + staging database) could be set up to run Cypress tests against a deployed instance without affecting production data.

7. How to Run Tests

7.1 Prerequisites

  • Node.js 22+ installed
  • Docker Desktop installed and running (for E2E tests)
  • Run npm install in both backend/ and FrontEnd/

7.2 Backend Unit & Integration Tests

cd backend
npm test          # Run all 150 tests once
npm run test:watch # Watch mode (re-runs on file changes)
No database required. All backend tests mock the database connection. These tests run entirely in memory and complete in under 3 seconds.

7.3 Frontend Unit Tests

cd FrontEnd
npm test          # Run all 21 tests once
npm run test:watch # Watch mode
No browser or server required. Frontend unit tests use jsdom to simulate the browser environment.

7.4 Cypress E2E Tests

E2E tests require the full application stack to be running:

# Terminal 1 – Start the database
docker compose up -d db

# Terminal 2 – Start the backend
cd backend
npm run dev

# Terminal 3 – Start the frontend
cd FrontEnd
npm run dev

Then run Cypress:

cd FrontEnd

# Option A: Interactive mode (opens Cypress GUI – great for debugging)
npm run cypress:open

# Option B: Headless mode (runs in terminal – CI-friendly)
npm run cypress:run

8. Deployment Compatibility – Vercel & Render

Verdict: No issues. The test suite is fully compatible with Vercel (frontend) and Render (backend) deployments. No configuration changes are needed.

Why there are no conflicts

ConcernAnalysis
Test dependencies in production? All test packages (vitest, cypress, supertest, jsdom, @types/supertest) are listed under devDependencies in package.json. Both Vercel and Render run npm install --omit=dev (or equivalent) for production builds, so these packages are never installed in production.
Test files bundled into build? Test files live in __tests__/ and cypress/ directories. They are never imported by any application source code. Next.js only bundles files under app/, lib/, and public/. The backend only runs files under src/ (excluding __tests__/). Test files are completely ignored during builds.
Config files cause build errors? vitest.config.ts and cypress.config.ts are only read by their respective test runners. Neither Next.js nor the backend server loads these files. They have no effect on builds or runtime.
Build time affected? No. Since devDependencies are skipped, there is no additional install time. Test files are not compiled during next build or backend transpilation.
Vercel-specific Vercel runs next build which only processes app/ routes and their imports. Test directories are excluded by default.
Render-specific Render runs the build command specified in the dashboard (typically npm install && npm start). With NODE_ENV=production, devDependencies are skipped automatically.

Optional: Add tests to CI pipeline

If you want tests to run automatically before each deployment, you can add a build command like:

# Render build command (optional)
npm install && npm test && npm start

# Vercel: Add to vercel.json (optional)
{
  "buildCommand": "npm test && next build"
}

This is optional and not required for deployment to work.

9. Conclusion

The EcoCity Builder test suite provides comprehensive coverage across all three layers of the testing pyramid:

  • 143 unit tests verify the core game simulation engine, authentication module, and frontend utilities in isolation with no external dependencies.
  • 28 integration tests validate all API endpoints with mocked database connections, ensuring correct HTTP status codes, response shapes, Zod validation, and JWT authentication.
  • 56 E2E Cypress tests exercise the full user journey — from landing page through registration, login, dashboard, game creation, tile placement, turn resolution, and results viewing — in a real browser against the live application stack.

All test packages are isolated in devDependencies and all test files reside in dedicated directories (__tests__/, cypress/), ensuring zero impact on production deployments to Vercel and Render.

Total: 227 test cases across 13 files, using 3 frameworks (Vitest, Supertest, Cypress), covering unit, integration, and end-to-end testing layers.
